# Contributing to rocci

## Dev setup

All three OSes: install [rustup](https://rustup.rs) (the pinned toolchain in
`rust-toolchain.toml` installs automatically on first build), [uv](https://docs.astral.sh/uv/),
and [just](https://github.com/casey/just) (`cargo install just`). Then:

```
just setup     # uv sync + maturin develop --release + pre-commit install
just test      # fast suite
```

**Windows note**: the Rust MSVC toolchain requires the *MSVC Build Tools*
("Desktop development with C++" workload from the Visual Studio installer).

## Everyday commands

`justfile` is the single answer to "how do I run X" — CI calls the same
recipes. Highlights: `just test-all` (full matrix including a
`ROCCI_BACKEND=numpy` pass), `just lint`, `just fix`, `just typecheck`,
`just rust-test`, `just bench`.

## Backend override (debugging only)

`ROCCI_BACKEND=numpy` forces the pure-NumPy fallback kernel;
`ROCCI_BACKEND=rust` forces the Rust core (raises if the extension is
missing). This is a debugging/testing knob, not a user feature — the two
backends agree statistically but produce different RNG streams, so bands are
not bit-identical across backends at the same seed.

## Slow statistical suite

`pytest -m slow` runs the heavy statistical tests (cross-backend
distributional agreement at B=8000 and, from milestone M6 on, the
calibration suite). `just test` excludes them; `just test-all` and CI run
everything.

## Golden-master fixtures (`tests/fixtures/golden/`)

The fixtures are recorded outputs of the *validated paper implementation*
(`studroc_paper`). They are data, not code:

- **Precedence rule (spec §5.7)**: if a golden test disagrees with the spec
  appendix, the fixture wins — the appendix has a transcription bug. Never
  regenerate fixtures to make new code pass.
- Fixtures change **only** with a spec change, in a PR that explains the
  statistical delta.
- Regeneration: `just fixtures` (needs a `studroc_paper` checkout with its
  venv; see `scripts/record_golden_masters.py`). The recorder prints and
  writes provenance (`PROVENANCE.md`).
- `check-added-large-files` blocks big files in pre-commit; golden masters
  are added deliberately with `git add -f tests/fixtures/golden/<file>.npz`.

## Branch model & commits

Trunk-based. `main` is protected: every CI job is a required check, linear
history, no direct pushes. Short-lived feature branches, **squash merge**.
PR titles follow [Conventional Commits](https://www.conventionalcommits.org)
(the squash makes your PR title the commit message); a CI check enforces
this. `CHANGELOG.md` is generated by git-cliff at release time.

## Code guidelines

- Lint/format: ruff (checker *and* formatter). Type check: ty. Config lives
  in `pyproject.toml`; run `just fix` to apply autofixes.
- Suppressions: `# ty: ignore[rule]` with the rule spelled out, never bare.
- Google docstrings on every public object, each with a runnable
  `Examples:` block (doctests run in CI and cannot rot).
- Rust: `cargo clippy --all-targets -- -D warnings` with pedantic enabled
  (allow-list documented in `rust/Cargo.toml`); `unsafe` only at the FFI
  boundary with `// SAFETY:` comments.
- Comments are evergreen; no commented-out code; no TODOs without a linked
  issue number.

## Release flow (owner)

See spec §14.5: `just release-prep X.Y.Z` → release PR → annotated tag
`vX.Y.Z` on the merge commit → `wheels.yml` builds, smoke-tests, and
publishes via trusted publishing (manual approval on the `release`
environment). rc tags go to TestPyPI.
